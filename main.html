Build Conflicts in The Wild

Abstract:

When collaborating, developers create and change software artifacts without being aware of other team members work. While such independence is essential for non-small teams increasing development productivity, it might also result in conflicts when integrating contributions. To better understand part of these conflicts— those revealed by failures when building integrated code— we investigate their frequency, structure, and adopted resolution patterns. We analyze merge scenarios from more than 500 Java projects that use Travis CI. To identify build conflicts, we select merge scenarios from git repositories, parse the Travis logs generated when building the commits in a scenario, and check whether the logged error messages are related to the integrated changes. We find and classify more than 640 build conflicts, and their resolution patterns. Most build conflicts are caused by missing declarations removed or renamed by one developer but referenced by the changes of another developer. These conflicts are often resolved by removing the dangling reference. We also identify build failures caused by immediate post integration changes, often performed with the aim of fixing merge conflicts. Based on the catalogue of build conflict causes derived, awareness tools could alert developers about the risk of a number of conflict situations. Program repair tools could benefit from the derived catalogue of build conflict resolution patterns, and the study infrastructure, to automatically fix conflicts. Thus, we developed a prototype that recommends to developers fixes for build conflicts.

Study Design

During this work, we design a study involving three main steps as illustrated bellow. The first step mines project repositories in GitHub and Travis for getting the information required for the experiment. The second step parses the Travis build logs of the collected scenarios to filter out scenarios with build breakages caused by non integration related causes such as execution timeout of the Travis services. Finally, the last step confirms conflict occurrence, and classifies conflicts and the adopted resolution patterns.
